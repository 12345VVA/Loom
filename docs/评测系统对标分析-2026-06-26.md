# workflow_eval 评测系统对标分析

> 日期：2026-06-26
> 范围：`backend/app/modules/workflow_eval/` 全模块（评估器、编排、数据模型、回归对比、前端）
> 目的：评估当前评测系统的实际能力边界，对标主流评测框架，找出不足与可提升点。

---

## 一、系统定性：它是什么

读完评估器、编排、模型、回归对比全链路后，本系统本质是一个**「工作流回归评测器（regression harness）」**——围绕"同一测试集 × 不同图版本，跑真实实例 + 规则/LLM 打分 + 按 case_key 对齐 diff"这件事，工程做得相当扎实。

有几处设计在同类系统里是**先进的**，先公允地肯定：

| 设计 | 评价 |
|---|---|
| [graph_json_snapshot](../backend/app/modules/workflow_eval/model/eval_run.py) 图快照 | 历史回归可比，不依赖 definition 当前版本——很多系统不做 |
| 每用例建真实 [WorkflowInstance](../backend/app/modules/workflow_eval/service/eval_orchestrator.py) 跑全链路 | 对工作流平台是正确选择（不是 mock 执行），能测到节点/governance/SSE 真实行为 |
| CAS 状态机 + [sweep 超时巡检](../backend/app/modules/workflow_eval/service/eval_orchestrator.py) + 失败兜底 | 健壮性达到生产级 |
| contextvar 精确 token/cost 关联 | 按工作流实例精确聚合，比 Promptfoo/LangSmith 的近似更准 |
| [EvaluatorRegistry](../backend/app/modules/workflow_eval/service/evaluator/base.py) 装饰器注册 | 插件机制本身是好的，扩展点已留好 |

**结论不是"系统差"，而是"作为一个 harness 很合格，但作为『评测系统』维度太窄、科学性不足"。** 下文分维度说明。

## 二、对标主流系统：差距地图

按相关度，最该对标的是这五类（不是所有都该学，但要知道差距在哪）：

- **Inspect AI**（UK AISI）—— agent/科研评测标杆
- **DeepEval**（Confident AI）—— NLG 工程化
- **Promptfoo** —— 对比矩阵 + CI 红队
- **LangSmith / Langfuse** —— 平台 + 在线/trace 评测
- **Ragas** —— 如果工作流含 RAG 节点

### 维度 1｜任务建模：只评估"终态输出"，不评估"过程"

**现状**：评估三元组是 `input → expected → actual(=workflow_output)`，`eval_tasks.py` 只取 `instance_result.get("output")` 喂给 evaluator。中间节点的输入/输出/分支决策完全不在评测视野内。

**主流**：Inspect AI 的 scorer 可挂在 solver chain 任意步；LangSmith 支持 **step-level feedback**（给 trace 某一节点打标）；Phoenix 评估整条 trace。

**差距**：一个工作流可能在中间节点就出错（比如检索召回烂、某 LLM 节点幻觉），但 final output 拼接后表面过关，evaluator 看不到。**这是工作流评测相比单 LLM 评测最该补、却最缺的能力。**

### 维度 2｜指标体系：只有 4 种规则 + 1 个裸 LLM judge

**现状**：`rule_match`（exact/contains/regex/numeric）+ `llm_judge`（单次 0-1 打分）+ `composite`（加权）。

**主流**：DeepEval 内置 20+ NLG 指标（faithfulness / answer-relevance / contextual-precision/recall），Ragas 把 RAG 链路指标拆细，lm-eval-harness 有 accuracy/CE/Brier/F1 等成熟指标。

**差距**：零领域标准指标。所有质量判断都压到一个泛化 LLM judge 上，用户只能靠手写 `evaluator_config` JSON 拼指标。

### 维度 3｜LLM Judge 的成熟度——最大的单项差距

**现状**（`llm_judge.py`）：

- 单次 pointwise 打分，返回 `float`；
- `_parse_score` **把 LLM 返回的 reason 直接丢弃**（只取 score），`judge_fn` 签名是 `→ float`；
- threshold 默认 0.6 硬编码；无偏置缓解。

**主流**：

- **G-Eval**：CoT + token 对数概率加权（不是只取最终数字）；
- **Pairwise comparison**：两两比较消除绝对分偏置（GPT-4 当 judge 对绝对分有位置/长度偏置是公认问题）；
- **Rubric-based**：多维 rubric 分别打分，而不是一个笼统 0-1；
- **Self-consistency**：多次采样投票降方差；
- **Judge 校准**：用 gold set 测 judge 与人类标注的 κ 一致性。

**差距**：本系统的 judge 是评测领域 2022 年的水平。而且 `reason` 被丢导致**无法做错误归因**——case 失败了只知道 score 低，不知道 judge 觉得哪里不对。

### 维度 4｜统计严谨性——科学性硬伤

**现状**：`finalize_eval_run` 算 avg/pass_rate/分位数；`regression.py` 用 **硬编码 0.1 阈值** 判退化/改善，纯点估 diff。

**主流**：Inspect 报置信区间 + bootstrap 重采样；DeepEval 在 ConfidentAI 报统计显著性；学术评测必报 n/CI/效应量。

**差距**：

- `delta=0.09` vs `0.1`：阈值附近完全是噪声（小样本尤其），但系统会给出"未退化"的确定结论；
- 无置信区间，无显著性检验（"新版真的更好还是抽到好 case？"答不了）；
- 回归对比 `compare.vue` 写死 `selectedRuns.length !== 2`，**只能 2 个 run 比，看不到多版本趋势**。

### 维度 5｜数据集管理：无版本、无分层、无切片

**现状**：`WorkflowTestSet` 无 version 字段；`tags` 字段存在但无 UI/无用于切片；导入仅 JSON 数组；case 无难度/类别/能力维度。

**主流**：LangSmith dataset 有 version + split（train/dev/test）+ 标签筛选；DeepEval dataset 版本化同步到平台；按能力维度（reasoning/coding/safety）切片聚合。

**差距**：**改一个用例，历史 run 的可复现性就受损**（没有数据集快照）。无法回答"在『多轮推理』子集上新版表现如何"这种切面问题。

### 维度 6｜可观测 / 在线评测 / 人工标注：基本为零

**现状**：纯离线、纯自动、无人工介入。token/cost 已补准。

**主流**：

- **在线评测**：LangSmith/Langfuse 从生产流量采样入 dataset，定期回放评测；
- **人工标注闭环**：annotation queue + 多标注者一致性（Cohen's κ / Fleiss' κ）+ gold set + disagreement review；
- **trace 评估**：见维度 1。

**差距**：无标注、无在线、无 trace 评估。对内容生成平台，**没有人工标注就无法校准 LLM judge 的可信度**——judge 准不准，没人知道。

### 维度 7｜自动化 / 迭代闭环：单向

**现状**：人手动起 run → 看表格 → 自己决策。

**主流**：

- **合成用例生成**：DeepEval 按能力自动生成对抗/边界用例；
- **自动优化**：DSPy/GEPA 自动调 prompt；
- **CI 门禁**：Promptfoo 在 PR 上跑断言，回归超阈值 block 合并；
- **失败归因**：失败 case 自动聚类。

**差距**：无合成数据、无 CI 门禁、无自动归因。评测结果不会自动反哺迭代。

### 维度 8｜鲁棒性 / 安全评测：零

**现状**：无对抗样本、无红队、无 prompt injection / 越狱 / PII 泄露 / 偏见检测。

**主流**：Promptfoo red-team（jailbreak/leakage/bias/PII 一键跑）、DeepEval 对抗生成。

**差距**：对一个 **AI 内容生成平台**，安全/鲁棒性评测缺席是战略级缺口——治理模块（governance）存在，却没有把它纳入评测维度。

### 维度 9｜实验可复现：半成品

**现状**：`graph_json_snapshot` + `version_label`（手填字符串）。

**主流**：完整 run manifest（seed/temperature/model version/profile prompt/evaluator 代码版本全量快照），一键复现。

**差距**：`version_label` 手填易错/易忘；**未快照 judge profile 的 prompt 和 evaluator 配置态**（profile prompt 改了，历史 run 的 judge 分就不可复现）；temperature/seed 等运行时参数未记录。

## 三、最该补的 5 个硬伤（不足，按危害排序）

| # | 硬伤 | 危害 | 位置 |
|---|---|---|---|
| 1 | **LLM judge 太朴素且丢 reason** | 判分不可信、无法归因，动摇整个质量评测的可信度 | `evaluator/llm_judge.py` |
| 2 | **回归 diff 无显著性/CI，阈值硬编码 0.1** | 小样本下结论是噪声；"新版更好"可能是错觉 | `service/regression.py` |
| 3 | **只评终态输出，不评 trace/中间节点** | 工作流特有的中间错误测不到，白做工作流评测 | `tasks/eval_tasks.py` |
| 4 | **数据集无版本/无切片** | 改用例即破坏历史可复现；无法按能力维度看 | `model/test_set.py` |
| 5 | **对比只支持 2 个 run + 无趋势** | 看不到多版本演进趋势，回归对比价值受限 | `views/run.vue` / `views/compare.vue` |

> 旁注：还有一些小瑕疵——`composite` 的 AND 语义（一不过全不过）会让加权 score 失去意义；rule 的 `actual[:200]` 与 judge 的 `_truncate 2000` 截断策略不一致。这些在补维度 1-3 时顺手修。

## 四、提升路线图（按 ROI 排序）

### 🟢 快赢（1–3 天，性价比最高）

1. **judge 保留 reason + 支持多维 rubric**：`judge_fn` 返回 `{score, dimensions: {...}, reason}`，存进 `evaluator_detail`，前端展示。立即可做错误归因。
2. **回归对比加 bootstrap 置信区间**：对 case-level score 做 1000 次重采样，`delta ± CI` 显示，CI 跨 0 即标"无显著变化"。`0.1` 阈值改为可配置 + 显著性双判定。
3. **多 run 趋势对比**：解除"只能选 2 个"限制，支持按 version_label 排序的趋势折线（pass_rate/avg_score/p95/cost 随版本演进）。
4. **数据集快照**：发起 run 时把当前用例集快照存进 `WorkflowEvalRun`（仿 graph_json_snapshot），保历史可复现。

### 🟡 中期（1–2 周，补能力维度）

5. **trace 级评估**：给 case_config 支持 `node_evaluators: {node_id: evaluator}`，在节点 flush 时按节点挂评估器——把"工作流评测"真正做出来。
6. **内置指标库**：补 faithfulness/answer-relevance/相似度/JSON-schema 校验这几个高频指标（可基于已有 `run_ai_chat`），减少用户手写 JSON。
7. **数据集切片**：用上 `tags` 字段，前端加标签筛选 + 按标签分组聚合指标。
8. **Pairwise judge + self-consistency**：用现有 `run_ai_chat` 实现 2-3 次采样投票 + 两两比较模式，缓解 judge 偏置。

### 🔴 战略（月级，构建闭环）

9. **人工标注 + judge 校准**：annotation queue（复用框架 CRUD）+ 用 gold set 算 judge 与人的 κ，把 κ 作为 judge 可信度指标显示出来。**这是让维度 3 从"能用"到"可信"的关键。**
10. **CI 门禁**：暴露一个无需登录的 `eval_run.start + 同步等待` 接口，配合 Promptfoo 风格的断言，让 PR 能跑回归门禁。
11. **安全/鲁棒性评测套件**：复用 governance 模块，做 prompt injection / PII / 偏见红队测试集——对内容平台是差异化竞争力。
12. **在线评测**：从生产 trace 采样入 dataset（已有 `AiModelCallLog` + instance 关联，基础设施齐了），定期回放。

## 五、总结

本系统**工程地基扎实（真实执行 + 图快照 + 精确 token 关联 + 健壮状态机）**，但**评测的科学性（统计显著性）、深度（trace 评估）、可信度（judge 校准 + 人工标注）三个维度明显落后于 Inspect AI / DeepEval / LangSmith 这一代系统**。

最快能拉升可信度的是路线图里的 **1（judge 保留 reason + rubric）、2（回归对比加 bootstrap 显著性）、9（人工标注 + judge 校准）** 三项。
