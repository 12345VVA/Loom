# Loom 框架待升级与补丁清单

> 生成日期：2026-04-20
> 基于对 `backend/app/framework`、`backend/app/modules`、`backend/app/core` 全部源码的逐文件审查。

---

## 一、安全类（高优先级）

### 1. 文件上传无校验 [已修复]

- **位置**：`app/framework/storage.py`、`app/modules/base/controller/admin/comm.py`
- **修复内容**：
  - 新增 `validate_upload()` 统一校验函数：扩展名白名单 + 文件大小限制
  - 新增 `UploadRejectedError` 异常类型，控制器层捕获并返回 400
  - `upload()` 和 `delete()` 均添加路径遍历防护（`os.path.abspath` + 前缀校验）
  - `config.py` 新增 `UPLOAD_MAX_SIZE_MB`（默认 10MB）和 `UPLOAD_ALLOWED_EXTENSIONS` 配置项

### 2. 无 API 限流 [已修复]

- **位置**：`app/framework/middleware/rate_limit.py`（新建）
- **修复内容**：
  - 新增 `RateLimitMiddleware` 全局中间件，基于 Redis 固定窗口计数器按 IP 限流
  - 按路由前缀分级限速：公开接口 30/min、管理端 60/min、其他 120/min
  - 白名单豁免（`/docs`、`/health` 等）
  - 超限返回 429 + `Retry-After` 响应头；正常请求附加 `X-RateLimit-Limit` / `X-RateLimit-Remaining`
  - Redis 不可用时自动降级为不限流，不影响业务
  - `config.py` 新增 `RATE_LIMIT_ENABLED`、`RATE_LIMIT_DEFAULT`、`RATE_LIMIT_ADMIN`、`RATE_LIMIT_OPEN`、`RATE_LIMIT_WHITELIST_PATHS` 配置项，可通过 .env 覆盖

### 3. CORS 过于宽松 [合理]

- **位置**：`backend/main.py:85-91`
- **问题**：`allow_methods=["*"]`、`allow_headers=["*"]`，攻击面过大
- **建议**：收紧到实际需要的 methods 和 headers

### 4. 密码存储算法 [合理]

- **位置**：`app/core/security.py`
- **问题**：使用 PBKDF2 + SHA256（10万次），可接受但不是最优
- **建议**：迁移到 `bcrypt` 或 `argon2`（抗 GPU/ASIC 攻击更强）

### 5. 默认管理员未强制改密 [合理]

- **位置**：`app/core/config.py`
- **问题**：`DEFAULT_ADMIN_USERNAME`/`PASSWORD` 从 .env 读取，首次登录无强制修改机制
- **建议**：首次登录时检测默认密码并强制修改

### 6. CSRF 防护 [合理]

- **位置**：全局
- **问题**：当前管理 API 全部依赖 Bearer Token，无 CSRF 风险；但如果未来引入 Cookie 认证需注意
- **建议**：暂不需要，但应在引入 Cookie 认证时同步添加 CSRF Token 或 SameSite 属性

## 二、架构 / 健壮性（高优先级）

### 7. 全局状态线程安全 [合理]

- **位置**：`app/framework/controller_meta.py:188-189`
- **问题**：`_registered_permission_configs` 和 `_registered_exported_routes` 是模块级全局列表，虽然启动时写入后只读，但缺少保护注释或显式冻结
- **建议**：启动完成后转为 `frozenset` 或加注释说明只读语义

### 8. 操作日志中间件同步写库 [合理]

- **位置**：`app/framework/middleware/operation_log.py:50-62`
- **问题**：在请求链中同步创建 Session 写入 `sys_log`，失败会吞异常且拖慢响应
- **建议**：改为 `BackgroundTasks` 异步写入，或用 Celery 任务

### 9. 响应包装中间件读全量 Body [合理]

- **位置**：`app/framework/middleware/response_envelope.py:23-28`
- **问题**：将整个响应体读入内存，大文件下载或大数据量场景下有内存风险
- **建议**：对 Content-Length 超阈值的响应跳过包装，或流式处理

### 10. 模块加载缺少错误隔离 [合理]

- **位置**：`app/modules/loader.py:34`、`loader.py:47`
- **问题**：模块导入失败、bootstrap 失败都会直接抛异常导致整个应用无法启动
- **建议**：catch 异常后标记为"加载失败"并记录日志，允许其他模块继续加载

### 11. BaseAdminCrudService 事务边界不清晰 [合理]

- **位置**：`app/modules/base/service/admin_service.py`
- **问题**：commit/refresh 分散在各方法内，缺少统一的事务管理；部分方法（如角色分配菜单后的权限缓存刷新）失败时缺少回滚补偿
- **建议**：引入 Unit of Work 模式或在控制器层统一管理事务边界

### 12. Session 泄漏风险 [合理]

- **位置**：`app/framework/middleware/admin_authority.py:36`、`scope_authority.py:40`、`operation_log.py:50`
- **问题**：直接 `Session(engine)` 创建会话，未使用 `with` 语句或依赖注入管理
- **建议**：统一使用 `with Session(engine) as session:` 或 `Depends(get_session)`

## 三、功能缺失（中优先级）

### 13. 无数据库迁移框架 [合理]

- **位置**：`app/core/database.py:51-54`
- **问题**：当前用 `create_all()` + 手工 ALTER TABLE 补字段（`database.py:70-135`），schema 演进无法追踪和回滚
- **建议**：引入 Alembic，将现有的 `_ensure_sqlite_compatible_schema` 逻辑迁入迁移脚本

### 14. EPS 元数据不完整 [合理]

- **位置**：`app/framework/eps/builder.py:55-57`
- **问题**：参数和响应类型硬编码为空 `"parameters": [], "responses": {}`；前端无法获得完整的类型提示
- **缺失项**：请求体 schema、响应体结构、枚举选项、验证规则的完整导出
- **建议**：结合 `scanner.py` 的模型扫描能力和 FastAPI 的 OpenAPI schema 补全 dts 字段

### 15. 无缓存失效策略 [合理]

- **位置**：`app/modules/base/service/cache_service.py`
- **问题**：用户角色/菜单变更后需手动刷新 Redis 缓存（`admin:perms:{userId}`），缺少基于事件的自动失效机制
- **建议**：在角色/菜单变更时自动清除相关用户的缓存键，或引入缓存标签机制

### 16. 文件存储仅本地 [合理]

- **位置**：`app/framework/storage.py`
- **问题**：只有 `LocalStorageProvider`，缺少 OSS/S3 实现；缺少分片上传、断点续传
- **建议**：实现 `OSSStorageProvider`，添加阿里云 OSS / AWS S3 支持

### 17. Redis 客户端过于简陋 [合理]

- **位置**：`app/core/redis.py`
- **问题**：`clear_cache_pattern()` 使用 `keys()` 命令，生产环境有性能风险（应改用 `SCAN`）；缺少连接池配置、哨兵/集群支持、健康检查
- **建议**：改用 `scan_iter()` 替代 `keys()`；配置 `connection_pool` 参数

### 18. 查询构造器 N+1 风险 [合理]

- **位置**：`app/framework/router/query_builder.py:116-134`
- **问题**：`apply_relations` 虽然用了 outerjoin，但 `_row_to_dict` 中访问关联对象属性仍可能触发懒加载
- **建议**：确保关联字段通过 `add_columns` 选取后不再访问 ORM 关系属性

### 19. CoolCache 装饰器功能单一 [合理]

- **位置**：`app/framework/cache.py`
- **问题**：不支持缓存空值防穿透、随机 TTL 防雪崩、手动失效
- **建议**：添加 `cache_empty=True`、`jitter=True`、`invalidate()` 等配置

## 四、生产就绪（中优先级）

## 六、新增漏洞与关注点（Antigravity 补充）

### 29. TaskInvoker 异步环境崩溃风险
- **位置**：`app/modules/task/service/task_invoker.py:137`
- **问题**：在已有的事件循环中调用 `asyncio.run()` 导致运行时报错。
- **影响**：系统任务触发时可能导致 worker 或 API 进程崩溃。
- **建议**：改为检测当前事件循环状态，或统一使用异步执行逻辑。

### 30. LocalStorageProvider 路径遍历漏洞
- **位置**：`app/framework/storage.py:54-56`
- **问题**：`delete` 方法仅通过简单替换获取路径。
- **风险**：攻击者可利用带有 `../` 的路径删除系统核心文件或敏感数据。
- **建议**：引入存储根路径校验（`os.path.abspath` 检查）。

### 31. JWT 密码版本字段大小写混用
- **位置**：`security_service.py`
- **问题**：`password_version` 与 `passwordVersion` 在代码库中交替出现。
- **风险**：虽然目前有兼容逻辑，但若未来重构遗漏某处可能导致所有 Token 校验失败。
- **建议**：在 `create_token` 和统一校验处规范化为单个字段名。

### 32. Redis 生产阻塞风险
- **位置**：`app/core/redis.py:24`
- **问题**：使用 `keys()` 命令匹配模式。
- **严重性**：在缓存量大时会造成 Redis 单线程长时间阻塞，导致全系统超时。
- **建议**：替换为 `scan_iter()`。

## 七、登录与认证安全（新增）

### 33. 缺少密码强度校验 [已修复]

- **位置**：`app/modules/base/service/auth_service.py:282-294`、`admin_service.py:357-371`
- **问题**：用户修改密码时未验证密码强度（长度、复杂度、常见弱密码检测）。
- **风险**：用户可设置弱密码如 "123456"，易被暴力破解。
- **修复内容**：
  - 在 `app/core/security.py` 中新增 `validate_password_strength()` 函数
  - 密码要求：至少8位，包含大小写字母、数字、特殊字符中的至少3种
  - 集成30+个常见弱密码黑名单（如 123456、password、qwerty 等）
  - 在用户自行修改密码和管理员添加/修改用户密码时均进行校验
  - 校验失败返回明确的错误提示信息

### 34. 验证码可被重放攻击 [已修复]

- **位置**：`app/modules/base/service/auth_service.py:237-244`
- **问题**：验证码校验后使用 `cache_delete()` 删除，但在校验失败时未删除，且无防重放机制。
- **风险**：同一验证码可被多次尝试，降低暴力破解难度。
- **修复内容**：
  - 获取缓存中的验证码后立即删除（`cache_delete()`）
  - 确保验证码只能使用一次，无论校验成功或失败
  - 更新错误提示为"验证码不正确或已失效"

### 35. 登录失败日志未记录设备指纹
- **位置**：`app/modules/base/model/sys.py:42-58`
- **问题**：登录日志虽记录 `device_id` 但为可选字段，登录失败时未强制要求设备指纹。
- **风险**：难以追踪同一设备的多次恶意登录尝试。
- **建议**：前端强制发送设备指纹，后端校验时记录并分析异常登录模式。

### 36. 无会话并发控制
- **位置**：`app/modules/base/service/authority_service.py:255`
- **问题**：单点登录（SSO）仅依赖缓存且可配置关闭（`ADMIN_SSO_ENABLED=False`），无强制单设备登录。
- **风险**：同一账号可同时在多设备登录，增加凭证泄露风险。
- **建议**：默认启用 SSO，提供会话管理接口让用户查看和踢出其他活跃会话。

### 37. Token 无吊销黑名单机制 [已修复]

- **位置**：`app/core/security.py:106-150`、`authority_service.py:231-280`、`auth_service.py:208-223`
- **问题**：Token 仅依赖过期时间和密码版本，无主动吊销机制（如强制登出、封号）。
- **风险**：管理员封禁用户后，已签发的 Token 在过期前仍可使用。
- **修复内容**：
  - **单Token级别黑名单**：基于JWT的jti字段，将特定Token加入Redis黑名单，TTL自动设置为Token过期时间
  - **用户级Token版本号**：引入 `token_version` 机制，递增版本号可使用户所有Token失效
  - **Token验证流程增强**：在 `get_user_from_access_token` 中依次检查：jti黑名单 → token_version → password_version
  - **用户登出**：调用 `logout` 时将当前Token的jti加入黑名单
  - **用户禁用**：管理员禁用用户时递增其token_version，使所有Token失效
  - **用户删除**：管理员删除用户时递增其token_version，确保无法继续使用
  - **向后兼容**：旧Token（无token_version字段）默认为0，不影响现有用户

### 45. 用户名与密码未去除前后空格 [已修复]
- **位置**：`app/modules/base/model/auth.py:107-116`、`251-274`、`276-299`、`161-173`
- **问题**：登录、创建用户、修改密码时未对用户名和密码进行 trim 处理。
- **风险**：
  - `"admin"` 和 `" admin"` 被视为不同用户，可绕过唯一性约束
  - 用户无意输入带空格的密码后无法登录（如复制粘贴时带入空格）
  - 可能被利用进行用户名欺骗攻击
- **修复内容**：
  - 在 `LoginRequest` 中添加 `field_validator` 对 username 和 password 自动 strip
  - 在 `UserCreateRequest` 中添加 strip 处理
  - 在 `UserUpdateRequest` 和 `UserPersonUpdateRequest` 中对 password 字段 strip
  - 在 `bootstrap_defaults()` 创建默认管理员时对配置值 strip

## 八、日志管理安全（新增）

### 38. 敏感操作未记录详细变更 [已修复]
- **位置**：`app/framework/middleware/operation_log.py:10-63`、`admin_service.py:225-378`
- **问题**：仅记录请求路径和参数，未记录变更前后数据对比（diff）。
- **风险**：审计日志无法追溯"谁在何时将数据从A改成了B"，合规性问题。
- **修复内容**：
  - 在 `admin_service.py` 中新增 `compute_entity_diff()` 和 `entity_to_dict()` 辅助函数
  - 修改 `update()` 方法：记录变更前的数据，计算变更差异，将diff记录到操作日志
  - 修改 `delete()` 方法：记录删除前的完整数据，保存到操作日志
  - 新增 `_log_entity_change()` 方法：统一处理实体变更日志记录
  - diff格式：`{"field": {"old": old_value, "new": new_value}}`

### 39. 日志写入失败被静默吞掉 [已修复]
- **位置**：`app/modules/base/service/auth_service.py:406-432`、`operation_log.py:141-156`
- **问题**：登录日志和操作日志写入异常时被 `except: return` 静默处理。
- **风险**：数据库故障时安全事件无记录，无法事后溯源。
- **修复内容**：
  - `_record_login_log()` 中的异常处理改为记录到Python logging系统
  - `operation_log.py` 中添加 `logger` 实例，写入失败时记录错误日志
  - 使用 `logger.error()` 配合 `exc_info` 参数记录完整堆栈信息
  - 日志写入失败不影响主业务流程（仅记录警告）

### 40. 日志表无自动清理机制 [已修复]
- **位置**：`app/modules/task/tasks/system_tasks.py`、`celery_app.py`
- **问题**：`sys_log` 和 `sys_login_log` 表虽配置了 `logKeep` 参数，但无定时清理任务。
- **风险**：日志无限累积导致磁盘占满、数据库性能下降。
- **修复内容**：
  - 新增 Celery 定时任务 `clean_expired_logs`，使用 SQLAlchemy 批量删除
  - 从 `SysParam` 表读取 `logKeep` 配置，默认7天
  - 使用 `crontab(hour=2, minute=0)` 设置为每天凌晨2点执行
  - 同时清理 `sys_log` 和 `sys_login_log` 两张表
  - 记录清理结果到日志系统（删除条数、保留天数等）
  - 在 `celery_app.py` 中配置 `beat_schedule` 启用定时调度

### 41. 日志脱敏不完整 [已修复]
- **位置**：`app/framework/middleware/operation_log.py:1-156`
- **问题**：仅脱敏 `password` 字段，未处理其他敏感字段（如手机号、身份证、银行卡）。
- **风险**：敏感个人信息在日志系统中明文存储，违反数据保护法规。
- **修复内容**：
  - 定义全局 `SENSITIVE_FIELDS` 集合，包含30+敏感字段名
  - 新增 `mask_sensitive_data()` 函数递归处理嵌套字典和数组
  - 分级脱敏策略：
    - 密码类：完全隐藏（`******`）
    - 手机号：保留前3后4位（`138****5678`）
    - 身份证：保留前6后4位
    - 银行卡：保留前4后4位
  - 在中间件读取Body后自动调用脱敏函数
  - 使用 `ensure_ascii=False` 保持中文可读性

## 九、密码管理增强（新增）

### 42. 默认管理员密码未强制首登修改 [已修复]
- **位置**：`app/modules/base/model/auth.py:58-78`、`auth_service.py:140-196`、`admin_service.py:565-572`
- **问题**：从 `.env` 读取的默认管理员密码无首登强制修改标识。
- **风险**：生产环境可能使用弱默认密码，且长期未修改。
- **修复内容**：
  - 在 `User` 模型中新增 `password_changed_at` 字段（Optional[datetime]）
  - 登录时检查该字段：`password_changed_at is None` 表示首次登录或从未修改过密码
  - 登录响应中添加 `forcePasswordChange` 标志，前端根据此标志引导用户修改密码
  - 用户自行修改密码时（`person_update`）更新 `password_changed_at` 为当前时间
  - 管理员重置密码时（`_before_update`）同样更新 `password_changed_at`
  - 新创建用户的 `password_changed_at` 默认为 `None`，强制首次登录修改密码

### 43. 密码重置无旧密码验证绕过风险 [已修复]

- **位置**：`app/modules/base/model/sys.py:62-93`、`sys_manage_service.py:352-486`、`admin_service.py:420-760`、`controller/admin/sys/security_log.py`
- **问题**：管理员可直接重置用户密码，但此操作未记录"谁重置了谁的密码"。
- **风险**：内部人员可滥用权限重置他人密码。
- **修复内容**：
  - **新增独立安全审计表** `SysSecurityLog`：与普通操作日志分离，专门存储敏感操作
  - **表设计特点**：
    - 记录操作者信息（ID、用户名、IP）
    - 记录操作对象信息（类型、ID、名称）
    - 记录变更详情（旧值、新值、diff）
    - 支持审计备注（操作理由）
    - 独立保留策略（不参与普通日志清理）
  - **新增 `SysSecurityLogService`**：提供专门的审计日志服务
  - **新增 `SysSecurityLogController`**：提供审计日志查询接口（只读）
  - **集成到关键服务**：
    - `UserAdminService`：记录用户创建/删除/禁用/密码重置
    - `RoleAdminService`：记录角色创建/删除/权限调整
  - **覆盖的操作类型**：
    - 用户管理：create/delete/disable_user/enable_user/reset_password
    - 角色管理：create/delete/update/update_permissions/update_data_scope
  - **查询友好**：按操作者、目标类型、操作类型、模块等多维度索引
  - **合规支持**：满足等保、ISO27001 对独立安全审计的要求

### 44. 密码哈希算法可升级
- **位置**：`app/core/security.py:19-42`
- **问题**：使用 PBKDF2-SHA256（10万次），虽安全但非抗 GPU/ASIC 最优解。
- **建议**：迁移到 `argon2`（内存硬化，抗 ASIC），保持向后兼容：用户登录成功后用 argon2 重新哈希。
