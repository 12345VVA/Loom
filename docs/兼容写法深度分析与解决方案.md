# 兼容写法深度分析与解决方案

> 生成日期：2026-04-20  
> 当前状态：本文是字段映射整改的历史分析与方案记录。主干实现已经统一为：模型与 Service 内部使用 `snake_case`，Pydantic 模型通过 `alias_generator=resolve_alias` 接受/输出 Loom 字段别名，EPS 中 `source` 保留后端字段，`prop/propertyName` 输出前端字段名。

---

## 一、问题根源

Loom-vue 前端使用 **camelCase** 命名，Python 后端使用 **snake_case** 命名。
早期项目缺少统一转换层，导致每个服务类都手动做双向映射，分散在约 **30+ 处**代码中。当前主干已改为以 `app/framework/api/naming.py` 的 `resolve_alias()` 为统一出口。

## 二、当前问题的三个层次

### 层次 A：请求入参（前端 → 后端）

前端发 camelCase，后端模型需要 snake_case 写入数据库。

**早期做法**：在 Pydantic Request 模型中使用 `AliasChoices` 接受两种命名，再在 Service 的 `_before_add` / `_before_update` 中手动 `pop` 转换。

```python
# auth.py — Request 模型声明双路接受
class MenuCreateRequest(BaseModel):
    parentId: Optional[int] = PydanticField(validation_alias=AliasChoices("parentId", "parent_id"))
    viewPath: Optional[str] = PydanticField(validation_alias=AliasChoices("viewPath", "component"))
    orderNum: int = PydanticField(validation_alias=AliasChoices("orderNum", "sort_order"))
    ...

# admin_service.py — 手动转换
def _before_add(self, data: dict) -> dict:
    data["parent_id"] = data.pop("parentId", None)
    data["path"] = data.pop("router", None)
    data["component"] = data.pop("viewPath", None)
    data["sort_order"] = data.pop("orderNum", 0)
    ...
```

**涉及文件**：
- `auth.py` — `UserCreateRequest`、`UserUpdateRequest`、`MenuCreateRequest`、`MenuUpdateRequest`、`DepartmentCreateRequest`
- `admin_service.py` — `UserAdminService._before_add/update`、`MenuAdminService._before_add/update`、`DepartmentAdminService._before_add/update`

**问题**：每个资源的每个字段都要写两遍（AliasChoices 声明 + 手动 pop 转换），新增字段时极易遗漏。

### 层次 B：响应出参（后端 → 前端）

数据库存 snake_case，前端期望 camelCase。

**现状做法**：在 `_row_to_dict` 中逐字段 `setdefault` 附加驼峰名，以及在 Pydantic Response 模型中直接用 camelCase 字段名。

```python
# admin_service.py 基类 — 通用字段
data.setdefault("createTime", data["created_at"])
data.setdefault("updateTime", data["updated_at"])
data.setdefault("status", 1 if data["is_active"] else 0)

# UserAdminService — 用户特有字段
data["name"] = data.get("full_name") or data.get("username", "")
data["nickName"] = data.get("nick_name") or ...
data["headImg"] = data.get("head_img")
data["departmentId"] = data.get("department_id")

# MenuAdminService — 菜单特有字段
data["parentId"] = data.get("parent_id")
data["router"] = data.get("path")
data["viewPath"] = data.get("component")
...
```

**涉及文件**：
- `admin_service.py` — `BaseAdminCrudService._row_to_dict`、`UserAdminService._row_to_dict`、`RoleAdminService._row_to_dict`、`DepartmentAdminService._row_to_dict`、`MenuAdminService._row_to_dict`

**问题**：`_row_to_dict` 中的字段映射是硬编码的，每个 Service 子类都要维护自己的映射表。且 `_row_to_dict` 返回的是 `dict`，绕过了 Pydantic 的类型校验。

### 层次 C：语义映射（不是简单的命名转换）

部分字段的 camelCase 和 snake_case 之间不是简单的命名转换，而是**语义不同**：

| 数据库字段 (snake_case) | 前端字段 (camelCase) | 语义差异 |
|------------------------|---------------------|---------|
| `parent_id` | `parentId` | 纯命名转换 |
| `sort_order` | `orderNum` | 字段名完全不同 |
| `is_active` | `status` (0/1) | 类型不同 (bool → int) |
| `is_show` | `isShow` | 纯命名转换 |
| `keep_alive` | `keepAlive` | 纯命名转换 |
| `path` | `router` | 字段名完全不同 |
| `component` | `viewPath` | 字段名完全不同 |
| `permission` | `perms` | 字段名完全不同 |
| `full_name` | `name` | 字段名完全不同 |
| `nick_name` | `nickName` | 纯命名转换 |
| `head_img` | `headImg` | 纯命名转换 |
| `department_id` | `departmentId` | 纯命名转换 |
| `password_version` | `passwordVersion` | 纯命名转换 |
| `created_at` | `createTime` | 字段名完全不同 |
| `updated_at` | `updateTime` | 字段名完全不同 |
| `delete_time` | `deleteTime` | 字段名完全不同 |

## 三、解决方案

### 方案：Pydantic AliasGenerator + 语义映射注册表

核心思路：**用 Pydantic v2 的 `AliasGenerator` 处理纯命名转换，用显式的字段映射表处理语义差异，彻底消灭手动 pop/setdefault。**

#### 3.1 定义统一的命名转换工具

```python
# app/framework/api/naming.py
"""统一命名转换：snake_case ↔ camelCase"""

def to_camel(name: str) -> str:
    """snake_case → camelCase"""
    components = name.split("_")
    return components[0] + "".join(x.title() for x in components[1:])

# 语义差异字段的全局映射表
# key = 数据库字段名 (snake_case)
# value = 前端字段名 (camelCase 或其他)
FIELD_ALIASES: dict[str, str] = {
    "sort_order": "orderNum",
    "is_active": "status",
    "path": "router",
    "component": "viewPath",
    "permission": "perms",
    "full_name": "name",
    "created_at": "createTime",
    "updated_at": "updateTime",
    "delete_time": "deleteTime",
}

def resolve_alias(field_name: str) -> str:
    """解析字段别名：优先使用语义映射，否则 camelCase 转换"""
    return FIELD_ALIASES.get(field_name, to_camel(field_name))
```

#### 3.2 在 Pydantic 模型中使用 AliasGenerator

**Request 模型**（入参）— 使用 `validation_alias` 接受前端的 camelCase：

```python
# 之前：每个字段手动写 AliasChoices
class MenuCreateRequest(BaseModel):
    parentId: Optional[int] = PydanticField(validation_alias=AliasChoices("parentId", "parent_id"))
    viewPath: Optional[str] = PydanticField(validation_alias=AliasChoices("viewPath", "component"))
    orderNum: int = PydanticField(validation_alias=AliasChoices("orderNum", "sort_order"))

# 之后：统一 AliasGenerator 自动处理
class MenuCreateRequest(BaseModel):
    model_config = ConfigDict(
        populate_by_name=True,
        alias_generator=resolve_alias,  # 自动映射
    )
    parent_id: Optional[int] = None     # 自动接受 parentId 和 parent_id
    path: Optional[str] = None          # 自动接受 router 和 path
    sort_order: int = 0                 # 自动接受 orderNum 和 sort_order
```

**Response 模型**（出参）— 使用 `serialization_alias` 输出 camelCase：

```python
# 之前：字段直接用 camelCase 命名，手动构造
class MenuRead(BaseModel):
    parentId: Optional[int] = None
    router: Optional[str] = None
    viewPath: Optional[str] = None
    orderNum: int

# 之后：字段用 snake_case，自动输出 camelCase
class MenuRead(BaseModel):
    model_config = ConfigDict(
        from_attributes=True,
        populate_by_name=True,
        alias_generator=resolve_alias,
    )
    parent_id: Optional[int] = None       # 序列化为 parentId
    path: Optional[str] = None            # 序列化为 router
    component: Optional[str] = None       # 序列化为 viewPath
    sort_order: int                       # 序列化为 orderNum
```

#### 3.3 类型转换也移至模型层

当前 `is_active` (bool) ↔ `status` (0/1) 的转换仍散落在 Service 的 `_before_add`/`_before_update`/`_row_to_dict` 中。应利用 Pydantic v2 的 `@field_validator` 和 `@field_serializer` 将类型转换统一收归模型层：

```python
# Request 模型 — 输入时 status: int → is_active: bool
class MenuCreateRequest(BaseModel):
    model_config = ConfigDict(populate_by_name=True, alias_generator=resolve_alias)

    is_active: bool = True

    @field_validator("is_active", mode="before")
    @classmethod
    def parse_status(cls, v):
        if isinstance(v, int):
            return v == 1
        return bool(v)

# Response 模型 — 输出时 is_active: bool → status: int
class MenuRead(BaseModel):
    model_config = ConfigDict(
        from_attributes=True,
        populate_by_name=True,
        alias_generator=resolve_alias,
    )
    is_active: bool = True

    @field_serializer("is_active")
    def serialize_status(self, v: bool) -> int:
        return 1 if v else 0
```

**原则**：全局通用的转换（如 `is_active` ↔ `status`）放在模型层；资源特有的语义转换（如菜单 `type` 的 `"group"/"menu"/"button"` ↔ `0/1/2`）仍在各自 Service 中标注为"语义转换"。

#### 3.4 消灭 Service 中的手动转换

AliasGenerator + validator/serializer 就位后，Service 中不再需要字段名 pop 和类型转换：

```python
# 之前：MenuAdminService._before_add（10 行映射 + 2 行业务）
def _before_add(self, data: dict) -> dict:
    data["parent_id"] = data.pop("parentId", None)
    data["path"] = data.pop("router", None)
    data["component"] = data.pop("viewPath", None)
    data["keep_alive"] = data.pop("keepAlive", True)
    data["is_show"] = data.pop("isShow", True)
    data["permission"] = data.pop("perms", None)
    data["sort_order"] = data.pop("orderNum", 0)
    data["is_active"] = int(data.pop("status", 1)) == 1
    data["type"] = self._normalize_menu_type(data.get("type"))
    return data

# 之后：仅保留纯业务逻辑
def _before_add(self, data: dict) -> dict:
    data["type"] = self._normalize_menu_type(data.get("type"))
    return data
```

#### 3.5 消灭 `_row_to_dict` 中的字段映射

```python
# 之前：BaseAdminCrudService._row_to_dict
data.setdefault("createTime", data["created_at"])
data.setdefault("updateTime", data["updated_at"])
data.setdefault("deleteTime", data["delete_time"])
data.setdefault("status", 1 if data["is_active"] else 0)

# 之后：直接返回 Response 模型实例，让 Pydantic 的 alias_generator + serializer 处理
def _to_response(self, row, response_model):
    data = self._raw_to_dict(row)  # 纯数据提取，不做映射
    return response_model.model_validate(data)
```

#### 3.6 性能评估

`AliasGenerator` 在每次序列化/反序列化时查一次映射表（Python dict O(1) 查找，单次约 50-100ns）。

量化估算（分页查询返回 200 条记录，每条 15 个字段）：

```
200 × 15 × 100ns ≈ 0.3ms
```

对比一次典型数据库查询（5-50ms），此开销完全可忽略。`FIELD_ALIASES` 使用模块级常量字典，不会动态构建，无额外开销。

## 五、收益

| 维度 | 改造前 | 改造后 |
|------|--------|--------|
| 字段映射代码行数 | ~150 行 | ~20 行（naming.py 映射表） |
| 新增字段时需改动处 | Request 模型 + Service 的 before_add + Service 的 before_update + _row_to_dict = 4 处 | Request/Response 模型 = 1 处 |
| 遗漏风险 | 高（手动 pop 容易漏字段） | 低（Pydantic 自动处理） |
| 语义映射表 | 无，散落在各 Service | 集中在 naming.py，一目了然 |
| 类型安全 | _row_to_dict 返回 dict，无类型校验 | Response 模型提供完整类型约束 |

## 六、风险与回退策略

- **风险**：前端依赖的某些字段名可能在映射表中遗漏
- **回退**：Pydantic v2 的 `populate_by_name=True` 确保两种命名都能接受，不会破坏现有接口
- **验证**：改造后对比 EPS 接口输出的 JSON 与改造前是否一致
