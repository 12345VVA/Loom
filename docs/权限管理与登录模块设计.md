# Loom 权限管理与登录模块设计

## 1. 当前目标

Loom 的权限体系已经从早期的“依赖式权限声明”演进为接近 Cool Admin 的管理端统一鉴权模型：

- 登录认证：`JWT Access Token + Refresh Token`
- 服务端状态：Redis 缓存优先，Redis 不可用时开发环境降级为进程内缓存
- 授权模型：`User -> Role -> Menu(permission)`
- URL 鉴权：权限点映射为 HTTP Method + URL pattern，由中间件统一拦截
- 数据权限：`Role.data_scope + RoleDepartmentLink` 解析为 `DataScopeContext`
- 前端菜单：后端动态菜单驱动路由与 `v-permission`

核心入口位于：

- `backend/app/framework/middleware/scope_authority.py`
- `backend/app/modules/base/service/authority_service.py`
- `backend/app/modules/base/service/auth_service.py`
- `backend/app/modules/base/service/data_scope_service.py`

## 2. 权限模型

### 2.1 主体与资源

- 用户：`sys_user`
- 角色：`sys_role`
- 菜单/权限资源：`sys_menu`
- 部门：`sys_department`

### 2.2 关系表

- 用户角色：`sys_user_role`
- 角色菜单：`sys_role_menu`
- 角色部门：`sys_role_department`

### 2.3 权限点格式

权限字符串统一使用：

```text
{module}:{resource}:{action}
```

示例：

- `base:sys:user:add`
- `base:sys:menu:export`
- `task:info:start`

`CoolControllerMeta` 生成的 CRUD 接口会自动登记权限点；自定义动作必须在 `@Get` / `@Post` 上显式声明同一个 `permission`，并与 `menu.json` 的按钮权限保持一致。

## 3. 登录链路

### 3.1 登录接口

```http
POST /admin/base/open/login
```

请求字段使用 Cool Admin 风格别名：

```json
{
  "username": "admin",
  "password": "Admin123456",
  "captchaId": "captcha-id",
  "verifyCode": "abcd"
}
```

响应字段保持前端兼容：

- `token`
- `refreshToken`
- `expire`
- `refreshExpire`
- `userInfo`
- `permission`

当默认管理员或新用户从未修改过密码时，`userInfo.forcePasswordChange` 为 `true`。

### 3.2 Token 载荷

Access Token / Refresh Token 统一包含：

- `sub`: 用户 ID
- `type`: `access` 或 `refresh`
- `userId`
- `username`
- `roleIds`
- `password_version`
- `token_version`
- `jti`

注意：Token 内部字段使用 `snake_case`，API JSON 响应通过字段映射输出 `passwordVersion` 等前端别名。

### 3.3 失效策略

- Token 到期自动失效
- 修改密码后 `password_version + 1`，旧 token 全部失效
- 登出时将当前 token 的 `jti` 加入黑名单
- 管理员禁用/删除用户或重置密码时递增用户级 `token_version`
- `ADMIN_SSO_ENABLED` 启用后，服务端缓存 token 与请求 token 不一致时拒绝访问
- `ADMIN_SESSION_MAX_CONCURRENT` 大于 0 时，只保留最近 N 个有效会话

## 4. 授权链路

管理端请求通过 `ScopeAuthorityMiddleware` 统一处理：

1. OPTIONS 预检直接放行
2. 根据 `/admin`、`/app`、`/aiapi` 判断 scope
3. 匹配 FastAPI route，读取 route tag 与 `permission`
4. 命中白名单或匿名 tag 时短路
5. 提取并校验 Bearer Token
6. 校验 token 黑名单、`password_version`、`token_version`、会话缓存
7. 超管直接放行
8. 普通用户按权限点和 URL pattern 检查
9. 当前用户与权限缓存写入 `request.state`

常用缓存键：

- `admin:token:{userId}`
- `admin:token:refresh:{userId}`
- `admin:perms:{userId}`
- `admin:permPatterns:{userId}`
- `admin:perms:path:{userId}`
- `admin:passwordVersion:{userId}`
- `admin:tokenVersion:{userId}`
- `admin:sessions:{userId}`
- `admin:department:{userId}`

## 5. 数据权限

CRUD 服务会通过 `BaseAdminCrudService._apply_query()` 自动调用 `resolve_data_scope()`，再将 `DataScopeContext` 传给 `QueryBuilder.apply_all()`。

支持的数据范围：

- `all`: 全部数据
- `self`: 仅本人数据
- `department`: 指定部门数据
- `department_and_children`: 指定部门及子部门数据

如果模型没有 `user_id` 或 `department_id` 字段，数据权限过滤会自动跳过。

## 6. 默认角色与权限

默认角色：

- `admin`：系统管理员，拥有全部权限与全量数据范围
- `task_operator`：任务操作员，拥有任务模块的常用操作权限

默认权限来源：

- 模块 `menu.json`
- `CoolControllerMeta` 自动 CRUD 权限
- 自定义路由装饰器上的 `permission`
- 模块配置中的兼容权限定义

## 7. 安全增强

当前已落地：

- 验证码一次性使用，防重放
- 密码强度校验
- 登录失败按账号和 IP 临时锁定
- 登录日志记录 IP、UA、客户端类型、设备指纹
- 密码哈希使用可配置 PBKDF2 迭代次数，并在登录成功后静默升级旧哈希
- 管理端变更请求可通过 `ADMIN_CSRF_ORIGIN_CHECK_ENABLED` 启用 Origin/Referer 校验
- 敏感操作记录到独立安全审计日志

## 8. 后续建议

- 若切换为 Cookie 认证，再补 CSRF Token 双提交机制
- 若要跨进程管理活跃会话，可把 `admin:sessions:{userId}` 扩展为可查询的会话表
- 业务模块新增敏感操作时，应接入 `SysSecurityLogService`
- 多租户能力稳定后，再把数据权限范围扩展到租户维度
