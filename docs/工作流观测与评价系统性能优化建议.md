## 一、现状判断

| 维度 | 现状 | 结论 |
|------|------|------|
| **观测** | 三层已就绪：实例级 `WorkflowInstance` + 节点级 `WorkflowExecutionLog`（[workflow.py:30-57](../backend/app/modules/workflow/model/workflow.py#L30-L57)）、Redis pub/sub 事件总线（[event_bus.py](../backend/app/modules/workflow/service/event_bus.py)）、SSE 端点（[instance.py:127-163](../backend/app/modules/workflow/controller/admin/instance.py#L127-L163)） | **框架齐全，但关键链路有性能债**：前端试运行未用 SSE 走轮询、统计全表内存聚合、SSE 订阅阻塞线程池、async 执行体内嵌同步 DB 写入 |

1. **脱敏是安全硬要求，不可削弱**：[audit S2](./workflow-audit-2026-06-22.md) 已确认 `WorkflowExecutionLog.input_data/output_data` 与 SSE 副本必须经 `AiSecurityService.mask_sensitive_dict` 脱敏后落库/推送。任何"把脱敏移出主路径"的优化都须保证**最终仍脱敏**，不得为性能牺牲合规。
2. **checkpointer 已持久化**：[audit S3](./workflow-audit-2026-06-22.md) 将默认后端从 `memory` 改为 `sqlite`（生产 `postgres`）。这意味着 LangGraph checkpointer **已是事实上的状态权威源**，`WorkflowExecutionLog` 仍全量存 `variables` 的冗余问题比此前更突出。

## 二、性能瓶颈诊断（分级）

### 🔴 P0-1　统计全表加载 + 纯 Python 聚合

**位置**：[stats_service.py:17-77](../backend/app/modules/ai/service/stats_service.py#L17-L77)

```python
logs = list(self.session.exec(statement).all())   # 整表 load 进内存
avg_latency = int(sum(item.latency_ms for item in logs) / total)   # 纯 Python 聚合
for item in logs: ...                              # 逐行分组
```

- **问题**：O(N) 内存 + CPU，每次请求全表扫描；`AiModelCallLog.created_at` 无索引 → 范围查询全表扫。日志涨到几十万行后，看板一次请求即可拖垮 Web 进程。
- **波及面**：该服务同时驱动 AI 看板（`/admin/ai/dashboard/cost`）与日志统计（`/admin/ai/log/stats`）。它也是未来工作流评价/回归看板的**通用聚合地基**——不改它，自建评价看板一上线就重蹈覆辙。

### 🔴 P0-2　`_async_execute` 事件循环被同步 DB / Redis 阻塞

**位置**：[workflow_tasks.py:174-199](../backend/app/modules/workflow/tasks/workflow_tasks.py#L174-L199)（节点循环内 `with Session(engine)` 写库）+ [event_bus.py:29-35](../backend/app/modules/workflow/service/event_bus.py#L29-L35)（`publish_event` 同步 `client.publish`）

- **问题（诊断成立）**：`_async_execute` 是 Celery Worker 内用 `asyncio.run()` 起的**独立事件循环**，里面跑 LangGraph 的真 async `astream`，却内嵌同步 `Session(engine)` 写库与同步 Redis publish。这个独立循环**不享受框架层的线程池卸载**（后者只作用于 FastAPI 路由），于是每次节点写库 + commit 都阻塞 worker 事件循环，节点多的工作流（循环/批处理）调度被 DB IO 串行拖慢。
- **处方须修正**：外部分析报告建议"全量上 AsyncSession"。**这与项目既定方向冲突**——`917d8da`/`f123537` 刚把框架路由统一改回同步 `def` + 线程池卸载，框架层依赖的是同步 `Session`。正确处方见[建议二](#建议二异步执行体内的阻塞段用-to_thread-卸载对齐项目既定方向)。

### 🔴 P0-3　SSE 订阅端用同步阻塞调用占用线程池

**位置**：[event_bus.py:55-67](../backend/app/modules/workflow/service/event_bus.py#L55-L67)

```python
message = await asyncio.to_thread(lambda: pubsub.get_message(timeout=1.0))
```

- **问题**：`get_message(timeout=1.0)` 是 redis-py 的**同步阻塞**调用，包在 `to_thread` 里 → **每条 SSE 连接独占一个 anyio 线程池 worker，最长阻塞 1 秒**。`ASYNC_THREAD_POOL_SIZE=100`（`917d8da`），并发监控连接稍多就会与其它 `to_thread` 调用争抢线程池，连带拖慢鉴权中间件等。
- **注**：外部分析报告只指出 `publish_event` 端的同步问题，**漏看了订阅端这个并发杀手**——订阅端才是长连接、占线程池的主力。

### 🔴 P0-4　前端试运行走轮询，未复用已建好的 SSE

**位置**：前端 [useWorkflowTest.ts:148-202](../frontend/src/modules/workflow/composables/useWorkflowTest.ts#L148-L202)（轮询）vs 后端 [instance.py:127-163](../backend/app/modules/workflow/controller/admin/instance.py#L127-L163)（已存在可用的 SSE `/stream`）

- **问题**：后端 SSE 基础设施已就绪并能逐节点实时推送，但前端试运行用 `info + logs` 轮询，间隔 2s→10s 递增；每次轮询 `logs` **全量拉取所有节点日志**。实时性最差到 10 秒延迟，且每轮两次 DB 查询。
- **性价比最高**：基础设施现成，只差前端接入，一次消灭"实时性差"与"服务端无谓负载"两个问题。架构说明第九节"试运行为轮询非真 SSE"这条已知局限，本质上就是这条。

### 🟠 P1-1　节点日志逐条同步写 + 全量快照冗余

**位置**：[workflow_tasks.py:174-199](../backend/app/modules/workflow/tasks/workflow_tasks.py#L174-L199)

- **问题**：每节点 = 开 Session → `session.get(instance)`（SELECT）→ 写 `state_data`（全量 100KB JSON）→ INSERT `exec_log`（`input_data` + `output_data` 各 100KB）→ commit，**4 次 DB 往返 + 两份近重复大 JSON**。
- **冗余强化**：checkpointer 持久化后（audit S3），`state_data` 每节点全量覆盖、`exec_log.input_data` 等于上一节点 `output_data`，皆为重复存储，单表体积膨胀，拖累后期统计查询。

### 🟠 P1-2　脱敏 + JSON 序列化重复且在主路径

**位置**：[workflow_tasks.py:193-208](../backend/app/modules/workflow/tasks/workflow_tasks.py#L193-L208)

- **问题**：对同一份 `new_vars` 做 `mask_sensitive_dict` + `json.dumps` **各两次**（落库一次、发事件一次）。含大文本/RAG 文档段时，O(N) 递归脱敏 + 序列化叠加在节点调度主路径上。
- **约束**：脱敏不可省（audit S2），只能挪位置、不能删。

### 🟠 P1-3　`logs` 接口无分页/增量

**位置**：[instance.py:108-125](../backend/app/modules/workflow/controller/admin/instance.py#L108-L125)

- **问题**：一次返回实例全部节点日志，配合 P0-4 的轮询被反复全量拉取；长流程/循环节点多的工作流 payload 膨胀。

### 🟡 P2-1　大载荷内联 JSON 落库（冷热未分离）

- **问题**：超大文本/图像 Base64 直接以内联 JSON 字符串进 `WorkflowExecutionLog`（单字段上限 100KB，[workflow.py:54-55](../backend/app/modules/workflow/model/workflow.py#L54-L55)），单表膨胀、统计查询全表扫描大字段。
- **出路**：项目已有 [framework/storage.py](../backend/app/framework/storage.py)（本地/S3-compatible），可存大载荷、DB 只留引用指针。

### 🟡 P2-2　索引缺口

- `AiModelCallLog.created_at` 无索引（P0-1 的范围扫描因此全表）。
- `WorkflowExecutionLog` 按 `instance_id` + `order by created_at` 查询，建议 `(instance_id, created_at)` 复合索引；现有 `node_id` 单列索引在该查询里未被使用。

## 三、优化建议

### 建议一（速赢·最高优先）：前端试运行接入 SSE + 统计聚合改造

这两条独立、收益大、不触碰执行链路，建议首先落地。

**A. 前端试运行接 SSE**（解决 P0-4）

- [useWorkflowTest.ts:148-202](../frontend/src/modules/workflow/composables/useWorkflowTest.ts#L148-L202) 的 `pollInstanceStatus` 改为订阅 `/stream`，由 `node_update`/`success`/`failed`/`paused`/`cancelled` 事件实时驱动 `updateNodeStatus`。
- `logs` 接口降级为**断线重连补偿 + 历史回填**（首连时拉一次，之后只接事件）。

**B. 统计聚合改 SQL + 缓存 + 索引**（解决 P0-1）

- `func.count()`/`func.sum()`/`func.avg()` + `group_by(func.date(created_at) / user_id / model_id)`，让数据库聚合，只回传分组结果。
- 加 Redis 缓存（TTL 5–15 min，key = `days`+`group_by`）。
- 给 `AiModelCallLog.created_at` 建索引。
- 进阶：每日 rollup 预聚合表（Celery Beat 凌晨汇总）。

> 这一条也是[评价系统](#四建设真正的评价系统)的**地基**——评价/回归必然涉及大批量历史聚合，先改它再起评价能力。

### 建议二（中期）：异步执行体内的阻塞段用 `to_thread` 卸载（对齐项目既定方向）

解决 P0-2 / P1-1 / P1-2，但**处方须与项目既定 async 策略一致**。

> 项目 `917d8da`/`f123537` 的既定方向：框架层"同步 `def` + `run_in_threadpool`/`asyncio.to_thread` 卸载"，**保留真 async 的只限确需 `await` 的场景**（file.read、SSE stream、aiapi stream）。本模块 `_async_execute` 与 SSE 订阅正属于"真 async 体内混了同步阻塞"的特例，应**就地卸载阻塞段**，而非全量迁移 AsyncSession。

**A. `_async_execute` 内 DB 写入卸载**（P0-2 + P1-1）

- 把"写 `state_data` + INSERT `exec_log` + commit"这一同步段，包进 `await asyncio.to_thread(_flush_node_log, ...)`（与 `917d8da` 在 `scope_authority` 里的 `asyncio.to_thread(_authorize)` 同型）。
- 进一步：exec_log 先攒到进程内 `asyncio.Queue`，由**后台协程**按"每 N 个节点 / 每 X 秒"批量 `bulk_save_objects`——实时性靠 Redis 事件保证，落库异步化。
- **不要**改框架层的同步 `Session` 依赖为 AsyncSession：工程量大、与 `BaseController`/`get_session` 不兼容、且逆 `917d8da` 方向。

**B. SSE 订阅改 `redis.asyncio`**（P0-3）

- `subscribe_events` 用原生异步 pubsub（`await pubsub.get_message(...)` 非阻塞），彻底脱离线程池。这与项目"真 async 保留"方向一致（对标 `aiapi stream_chat` 保留 async）。

**C. 脱敏 + 序列化移到后台协程**（P1-2，且不削弱安全）

- 主路径只把原始 `new_vars` `put_nowait` 进同一 `asyncio.Queue`；后台协程负责脱敏 + `json.dumps` + 批量落库。
- **同进程**传递对象引用合法（这是对外部报告"Celery 子任务传未序列化引用"方案的修正——跨进程不可传引用，进程内可以）。
- 安全保证：后台协程落库前**仍调** `mask_sensitive_dict`，最终落库/SSE 副本与 audit S2 要求一致，只是不再阻塞节点调度。

**D. 节点载荷 Diff 化**（P1-1）

- `exec_log.output_data` 只记**本节点产生的变更（Delta）**，完整状态回溯依赖已持久化的 checkpointer；或保留全量但按节点类型可选关闭（如循环节点只记汇总）。

### 建议三（中期）：`logs` 增量化 + 分页

解决 P1-3：增量参数 `?after_id=<last_id>`（配合 SSE 做补偿）或分页；加 `(instance_id, created_at)` 复合索引。

### 建议四（长期）：冷热分离 + 建设评价系统

**A. 冷热分离**（P2-1）：超大载荷经 [framework/storage.py](../backend/app/framework/storage.py) 落对象存储，`exec_log` 只存引用 URL。

**B. 评价系统**：见下节。

## 四、建设真正的评价系统

现状只有"测试观测"，无"自动评价"。两条出路，推荐组合：

### 出路 A（推荐）：集成 Langfuse / LangSmith

- LangGraph `compile()` 时插入第三方 Callback Handler，**一次性获得** Span 级 Tracing、Timeline UI、基于真实调用的评价反馈（打分、人工标注、测试集回归）。
- **选 Langfuse**：开源可自托管，与项目自托管定位契合；原生支持 LangGraph，异步打点不阻塞主路径——正好规避本报告前述"自建观测的性能陷阱"。
- 覆盖：Token 消耗、Span 耗时、LLM 质量打分、A/B 对比、回归测试。

### 出路 B（补充）：自建业务侧评价

若需留在平台内（不引外部依赖），补：

| 能力 | 落点 |
|------|------|
| 执行质量评分 | `WorkflowExecutionLog` 加 `score`/`eval_data`；终态节点后接 LLM-as-judge 或规则评估器 |
| 测试集 + 批量评估 | 新增 `WorkflowTestSet`/`WorkflowEvalRun`，提交测试集 → Celery 批量跑实例 → 汇总通过率/均分/P95 延迟/总 token 成本 |
| 回归对比 | 以 `graph_json` 版本为锚，A/B 对比两版本在同测试集上的指标差异 |
| 离线评估队列 | 复用 Celery 多队列架构，开独立 `workflow.eval` 队列，避免批量评估挤占实时 `workflow` 队列 |
| 指标看板 | 节点级 P50/P95 延迟、失败率、单节点成本——**前提是先完成建议一 B 的聚合改造** |

> **关键约束**：评价/回归必然涉及大批量历史数据聚合，所以**建议一 B（统计聚合改造）是评价系统的地基**。先做聚合改造，再起评价能力，否则一上线就被全表扫描拖垮。

  ├─ 大载荷冷热分离到对象存储（P2-1）

**若只动三处**，按此顺序：

3. **SSE 订阅换 `redis.asyncio`**（P0-3）——解锁高并发监控连接，避免线程池耗尽。

### 勘误（外部报告的技术性误判）

1. **"全量升级 AsyncSession" 与项目既定方向冲突**。
   `917d8da`/`f123537`（2026-06-25）已把框架路由统一改回同步 `def` + `run_in_threadpool`/`asyncio.to_thread` 卸载，框架层依赖同步 `Session`。全量 AsyncSession 工程量大、与 `BaseController`/`get_session` 不兼容、且逆既定方向。**正确处方**：在 `_async_execute` 独立事件循环内，对 DB 写入段用 `asyncio.to_thread` 卸载（与 `scope_authority` 同型）。

2. **"Celery 子任务传递未序列化的原始变量引用" 跨进程不成立**。
   Celery 跨进程序列化参数，Python 对象引用不跨进程。**正确处方**：进程内 `asyncio.Queue` + 后台协程脱敏落库（同进程可传引用）；如确需跨进程卸载 CPU，发送方必须先序列化，序列化成本仍在主路径，收益有限。

3. **"脱敏移出主路径"不得削弱 audit S2 的安全要求**。
   S2 已确认 `input_data/output_data` 与 SSE 副本必须脱敏。优化只能是"挪到后台协程、最终仍脱敏"，不能为性能省略脱敏步骤。

### 过时判断的更新

- **checkpointer 已持久化**（audit S3，默认 `sqlite`/生产 `postgres`）："状态全量落盘与 checkpointer 重复"这一判断**比初版更成立**，`WorkflowExecutionLog` 全量存 `variables` 的冗余应优先 Diff 化。
- **cancel / per-node 超时已具备**（audit S4）：`_async_execute` 已有 `asyncio.wait_for(__anext__, node_timeout)` 与协作式取消回查，本报告不再将其列为缺口，优化聚焦"写库阻塞"而非"无法取消"。
