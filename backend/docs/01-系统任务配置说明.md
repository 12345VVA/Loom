# 系统任务配置与自动化执行手册

本项目已实现类 Midway (`cool-admin`) 风格的任务动态调用机制。您可以通过后台管理界面的“任务管理”模块，配置任意模块中的 Service 方法作为定时任务执行。

## 1. 任务路径 (Service Path) 配置

在添加或修改任务时，**执行函数**（即 `service` 字段）请按照以下格式填写：

`{模块短名}:{方法名}`

### 命名约定：
- **模块短名**：默认为 `模块名.类名去掉Service后的部分` (转小写)。
    - 如果类名以模块名开头，则会自动去掉模块名部分（例如 `DictInfoService` 在 `dict` 模块中对应 `dict.info`）。
- **示例**：
    - 调用 `DictInfoService.list` 方法：`dict.info:list`
    - 调用 `TaskInfoService.stats` 方法：`task.info:stats`
    - 调用 `AuthService.test` 方法：`base.auth:test`

## 2. 编写可被调用的服务类

所有存放在 `app/modules/*/service/` 目录下，且类名以 `Service` 结尾的类都会被自动扫描。

### 方法结构要求：
1. **构造函数**：必须能接受 `session: Session` 参数。
2. **目标方法**：建议接受一个 `data` 参数用于接收配置的任务参数。

```python
# 示例：app/modules/demo/service/demo_hello_service.py
class DemoHelloService:
    def __init__(self, session):
        self.session = session

    def say_hi(self, data):
        # data 是解析后的 dict
        name = data.get("name", "Guest")
        return f"Successfully said hi to {name}"
```

## 3. 智能参数转换 (Smart Invocation)

为了方便复用现有的基础服务，执行器具备“智能签名识别”功能：

- **CrudQuery 自动转换**：如果目标方法（如 `list`, `page`）期望一个 `CrudQuery` 对象，执行器会自动将任务配置中的 JSON `data` 转换为 `CrudQuery` 实例，并将非通用参数填充到 `eq_filters` 中。
- **普通参数注入**：如果方法仅接收一个 `data` 参数，则直接透传 Python 字典。

## 4. 后台执行原理

1. **触发**：通过前端点击“执行一次”或通过 `celery-beat` 定时触发。
2. **分发**：执行 Celery 任务 `task.execute_system_task(task_id)`。
3. **实例化**：`TaskInvoker` 根据路径扫描模块，找到 Class，打开 DB Session 并实例化。
4. **日志**：执行结果（返回值或 Exception）会自动记录到 `task_log` 表中，包括耗时信息。

## 5. 调试建议

- 如果日志提示 `Service not found`，请检查类名是否以 `Service` 结尾，以及是否在 `service` 文件夹下。
- 使用 `dict.info:list` 作为填写的测试目标，如果执行日志显示了 JSON 数据，代表调用链已通。
